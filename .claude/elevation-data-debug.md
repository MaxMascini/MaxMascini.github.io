# Halifax Elevation Data — Debug State (2026-03-27)

## Problem

`public/data/halifax-elevation.json` was generated by the OpenTopoData point-query API.
That API interpolates over water bodies instead of returning void values.
Result: only 1.2% zeros — harbour, NW Arm, Bedford Basin all have land-like elevations.
The peninsula is invisible in the topo rendering.

## Solution Identified

Use the AWS Terrain Tiles SRTM1 "Skadi" file — **no auth, public S3 bucket**:

```
https://s3.amazonaws.com/elevation-tiles-prod/skadi/N44/N44W064.hgt.gz
```

- Single download (~3.5 MB compressed, ~24.7 MB decompressed)
- Standard SRTM1 format: **3601×3601 int16 big-endian** (NOT 1201×1201!)
- Ocean/NoData = -32768 → reliable water masking
- Tile covers 44–45°N, 63–64°W (Halifax peninsula is inside)

## The Bug in the Current Script

`scripts/fetch-halifax-elevation.py` has ONE wrong line:

```python
# WRONG — SRTM3 dimensions (90m resolution)
arr = np.frombuffer(raw, dtype=">i2").reshape(1201, 1201).astype(float)

# CORRECT — SRTM1 dimensions (30m resolution)
arr = np.frombuffer(raw, dtype=">i2").reshape(3601, 3601).astype(float)
```

That's the only change needed. Everything else in the script (crop bounds, normalization, output format) is correct.

## Verification That Data is Valid

First 20 bytes of decompressed file (hex): `001f002100220023002200210020001f001e001e`

As int16 big-endian: `[31, 33, 34, 35, 34, 33, 32, 31, 30, 30]` meters

These are the northwest corner of tile N44W064 (near Cape Breton approach / Nova Scotia mainland).
Values of 31–35m are geographically plausible land elevations. ✓

Ocean cells should be -32768 (SRTM void). The normalization code sets these to -1.0 so they sit below all contour levels, creating clear open water.

## Expected Result After Fix
